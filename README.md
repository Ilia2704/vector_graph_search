# vector_graph_search

Локальное demo для занятия:

**«Поиск в базе знаний: где векторы ошибаются, а графы помогают»**

Проект показывает, почему обычного vector search иногда недостаточно для RAG-приложений, и как граф знаний помогает улучшить retrieval.

В demo нет LLM, Ollama, OpenAI API, Neo4j или Docker.  
Цель проекта — показать именно слой retrieval:

```text
пользовательский запрос
→ naive vector search
→ неправильный или неполный контекст
→ graph lookup
→ filtered vector search
→ правильный evidence
```

Qdrant запускается локально в in-memory режиме.

---

# Бизнес-смысл demo

В реальном RAG-приложении пользователь обычно не хочет “самый похожий текст”.

Он хочет правильный ответ.

Но правильный ответ часто зависит не только от текста, а от:

- бизнес-правил;
- связей между сущностями;
- технических идентификаторов;
- исключений;
- признаков продукта;
- принадлежности сервиса к бизнес-функции.

Vector search хорошо ищет похожие фрагменты текста.  
Но он плохо понимает, что:

```text
“без мобильного интернета”
```

это не просто тема “мобильный интернет”, а строгое бизнес-условие.

И что:

```text
“Шлюз Госуслуг”
```

может называться в логах совсем иначе:

```text
esia-bridge-prod
```

Граф знаний нужен как слой структуры:

```text
Product -[INCLUDES]-> Feature
TechnicalService -[IMPLEMENTS]-> BusinessService
```

После этого Qdrant ищет не по всему корпусу, а только внутри правильной логической области.

# Что показывает проект

Проект содержит два business-oriented кейса.

---

## Case 9. Слепота к отрицанию: тариф без мобильного интернета

### Бизнес-сценарий

Представим, что мы строим RAG-помощника для сотрудников контакт-центра или онлайн-продаж.

Пользователь спрашивает:

```text
Посоветуй тарифы, которые строго НЕ включают мобильный интернет.
```

На первый взгляд это обычный semantic search.  
Но на самом деле здесь есть жесткое бизнес-условие:

```text
feature "Мобильный интернет" должна отсутствовать
```

Если система порекомендует тариф с мобильным интернетом, это будет не просто “неидеальный ответ”, а ошибка бизнес-логики.

---

### Что лежит в Qdrant

В Qdrant загружены три тарифных документа:

```text
Тариф Базовый
Тариф Смарт
Тариф Премиум
```

Смысл документов:

```text
Тариф Базовый:
- включает звонки и SMS;
- мобильный интернет не входит.

Тариф Смарт:
- включает звонки, SMS и безлимитный мобильный интернет.

Тариф Премиум:
- включает мобильный интернет в роуминге и приоритетную поддержку.
```

Правильный ответ:

```text
Тариф Базовый
```

---

### Что делает naive vector search

Naive vector search получает запрос:

```text
Посоветуй тарифы, которые строго НЕ включают мобильный интернет.
```

И возвращает:

```text
#1 Тариф Смарт
#2 Тариф Премиум
#3 Тариф Базовый
```

Это выглядит странно, но для vector search это нормально.

Почему?

Потому что запрос содержит сильный смысловой якорь:

```text
мобильный интернет
```

А тексты `Тариф Смарт` и `Тариф Премиум` явно содержат эту feature.  
Vector search находит похожие тексты, но не понимает, что пользователь просит эту feature исключить.

Иными словами, vector search отвечает на вопрос:

```text
Какие документы похожи на тему “мобильный интернет”?
```

А бизнес-вопрос был другим:

```text
Какие продукты НЕ имеют feature “Мобильный интернет”?
```

---

### Что знает граф

Граф хранит продуктовые связи:

```text
Тариф Смарт -[INCLUDES]-> Мобильный интернет
Тариф Премиум -[INCLUDES]-> Мобильный интернет
```

У `Тариф Базовый` нет такого ребра:

```text
Тариф Базовый -[INCLUDES]-> Мобильный интернет
```

Это означает, что `Тариф Базовый` не включает мобильный интернет.

---

### Как граф решает задачу

Графовая операция:

```text
predecessors(target='Мобильный интернет', relation='INCLUDES')
```

возвращает продукты, которые включают мобильный интернет:

```text
Тариф Смарт
Тариф Премиум
```

Дальше система делает логическое исключение:

```text
allowed_products = all_products - products_with_mobile_internet
```

Результат:

```text
Тариф Базовый
```

После этого Qdrant запускается повторно, но уже с фильтром:

```text
product == "Тариф Базовый"
```

И возвращает правильный документ.

---

### Вывод по Case 9

Vector search нашел похожие тексты.  
Graph search задал допустимое множество продуктов.  
Filtered Qdrant search вернул правильный evidence.

Главная идея:

```text
Если вопрос содержит отрицание или бизнес-ограничение,
нельзя полагаться только на similarity.
```

## Case 4. Бизнес-термин против технического ID

### Бизнес-сценарий

Представим внутреннего RAG-помощника для SRE, support или incident management.

Пользователь спрашивает:

```text
Какой сейчас error rate у Шлюза Госуслуг?
```

Для человека вопрос понятен.  
Но внутри компании разные системы могут использовать разные названия одной и той же сущности.

Бизнес и support говорят:

```text
Шлюз Госуслуг
```

А DevOps-логи используют техническое имя:

```text
esia-bridge-prod
```

Если RAG ищет только по текстовой похожести, он может найти документы, где явно встречается бизнес-термин, но пропустить лог, где лежит реальная метрика.

---

### Что лежит в Qdrant

В Qdrant загружены четыре документа:

```text
1. Runbook про Шлюз Госуслуг

2. CMDB-документ:
   esia-bridge-prod реализует бизнес-сервис "Шлюз Госуслуг"

3. DevOps Log:
   Service: esia-bridge-prod
   Metric: 5xx_ratio_percent = 15

4. DevOps Log:
   Service: kafka-billing-consumer
   Metric: consumer_lag = 0
```

Правильный evidence находится в документе:

```text
[DevOps Log] Service: esia-bridge-prod.
Metric: 5xx_ratio_percent = 15
```

---

### Что делает naive vector search

Пользователь спрашивает:

```text
Какой сейчас error rate у Шлюза Госуслуг?
```

Naive vector search возвращает:

```text
#1 Runbook про Шлюз Госуслуг
#2 CMDB про связь Шлюза Госуслуг и esia-bridge-prod
```

Эти документы похожи на запрос, потому что содержат фразу:

```text
Шлюз Госуслуг
```

Но они не содержат текущую метрику error rate.

Правильный лог написан техническим языком:

```text
esia-bridge-prod
```

Поэтому он может быть ниже по vector similarity, хотя именно он содержит правильный ответ.

---

### Что знает граф

Граф хранит связь между бизнес-сервисом и техническим сервисом:

```text
esia-bridge-prod -[IMPLEMENTS]-> Шлюз Госуслуг
```

Это означает:

```text
технический сервис esia-bridge-prod реализует бизнес-сервис "Шлюз Госуслуг"
```

---

### Как граф решает задачу

Графовая операция:

```text
predecessors(target='Шлюз Госуслуг', relation='IMPLEMENTS')
```

возвращает technical service ID:

```text
esia-bridge-prod
```

После этого Qdrant запускается повторно, но уже с фильтром:

```text
doc_type == "devops_log"
service_id == "esia-bridge-prod"
```

Теперь поиск идет не по всему корпусу, а только по operational logs нужного сервиса.

Результат:

```text
[DevOps Log] Service: esia-bridge-prod.
Status: CRITICAL.
Metric: 5xx_ratio_percent = 15 during the last 5 minutes.
```

---

### Вывод по Case 4

Naive vector search нашел похожие документы.  
Но graph lookup нашел правильную техническую сущность.  
После этого Qdrant достал правильный operational evidence.

Главная идея:

```text
Правильный документ может быть менее похожим текстом,
если пользователь говорит бизнес-языком,
а данные хранятся техническим языком.
```

# Архитектурная идея

В обоих кейсах Qdrant работает корректно.  
Проблема не в Qdrant.

Проблема в том, что naive vector search решает только одну задачу:

```text
найти ближайшие векторы
```

Но business retrieval часто требует другого:

```text
найти правильные сущности
применить правила
разрешить бизнес-термин в технический ID
исключить запрещенные варианты
ограничить область поиска
```

Поэтому итоговый pattern такой:

```text
1. Пользователь задает вопрос
2. Система извлекает сущности
3. Граф определяет связи и допустимые объекты
4. Qdrant ищет только внутри правильного подмножества
5. LLM получает более точный контекст
```

Коротко:

```text
Vector search = поиск похожего текста
Graph search = поиск связанных сущностей и правил
Graph-augmented RAG = vector search внутри правильного бизнес-контекста
```

# Почему demo без LLM

В этом проекте нет генерации ответа через LLM.

Это сделано специально.

Если добавить LLM, аудитория может начать обсуждать:

- качество промпта;
- стиль ответа;
- hallucinations;
- модель;
- latency;
- API-ключи.

Но цель demo другая:

```text
показать, как формируется контекст для LLM
```

Если retrieval достал неправильный контекст, LLM может красиво сформулировать неправильный ответ.

Поэтому demo изолирует retrieval layer:

```text
Qdrant без графа
vs
Qdrant + graph-derived filter
```

# Почему векторы заданы вручную

В demo используются маленькие фиксированные векторы:

```python
vector=[1.0, 1.0, 0.6, 0.1, 0.0]
```

В production эти векторы строятся embedding-моделью.

Здесь они заданы вручную, чтобы:

- demo было стабильным;
- не скачивать embedding model во время занятия;
- не зависеть от интернета;
- гарантированно показать retrieval failure;
- сфокусироваться на архитектуре, а не на конкретной embedding-модели.

Смысл demo от этого не меняется.

Qdrant все равно делает vector search:

```text
query vector → nearest vectors
```

А граф все равно добавляет структуру:

```text
entity → relation → entity
```
